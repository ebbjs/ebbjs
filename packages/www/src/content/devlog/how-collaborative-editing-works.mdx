Once you start down the local-first rabbit hole, it doesn't take long to encounter the term CRDT.

CRDT stands for **Conflict-Free Replicated Data Type**. These are a relatively new field of data structures that use algorithms to automatically merge divergent state without the need of an authoritative server.

On paper and in papers, they present as a kind of god-send; mathematical magic that lets developers avoid the most gnarly and complicated problem when building offline-calable software: conflict resolution.

Unfortunately though, the theory doesn't really hold up in practice. Not because the CRDT algorithms don't work correctly. But because conflict resolution is not a math problem, it's a social problem.

So, in this article, I'm going to explain why CRDTs don't actually make your data "conflict-free" and share why embracing conflicts instead of avoiding them is the best approach for most offline-first apps.

## Part 1: The Color of Magic

The Moment team recently [illustrated the problem with relying on CRDTs for conflict resolution](https://www.moment.dev/blog/lies-i-was-told-pt-1) with a deceptively simple test.

Let's say Alice and Bob are both offline editing a document that contains the heading:

> The Color of Magic

Bob is British, so he changes "Color" to "Colo**u**r". But Alice deems the heading uneccesarry and decides to delete it.

Later, they both come online and the CRDT used to model the document content automatically merges their changes. The resulting text?

> u

Just the letter "u". The character Bob added that wasn't one Alice deleted.

This is _technically_ correct according to most CRDT merge algorithms. But when the Moment team presented cases like this to _actual_ users, they found that roughly 20-30% of these automatically resolved conflicts produced results those users deemed as **data-corruption**.

The CRDT did exactly what it was designed to do: converge deterministically without coordination. But _how_ it converged those changes was not how either Alice or Bob would do it.

One way to interpret this example is that CRDTs need better automatic merge algorithms. We need to go back to the whiteboard and soften the edge cases.

The problem is that this "perfect algorithm" does not exist. Alice and Bob made edits to the document based on their current view of it. But they may (or may not) have made _different_ decisions given the context of the other's changes.

Alice might not have deleted the heading if she knew Bob was editing it. Bob might not have bothered with the typo if he knew Alice was removing the whole thing.

And there is no amount of math that can precisely predict the answer to these "what if" questions about user intent.

CRDTs aren't a silver bullet for dealing with conflicting offline writes because being able to _merge_ the data was never really the problem - it was operating on incomplete information in the face of concurrent edits.

CRDTs don't change that underlying issue. As the CTO of Convex put it in a podcast about the challenges of building local-first software,

> There's no amount of being smarter that can...

Once we internalize this and realize that allowing offline-writes makes conflicts unavoidable, we basically have two options:

1. Give up and go back to a strongly consistent, authoritative server model
2. Embrace that conflicts will happen and design our apps accordingly

I've chosen to build ebb for those who want to take option 2.

## Part 2: Confronting conflicts

So, what if we stopped trying to be smart enough to avoid conflicts and just embraced them?

And when you allow offline writes, that's a problem with a non-deterministic .

This is why most applications default to using a data layer that provides **strong consistency**. Because when writes all have to go through a central, authoritative server, concurrent edits are a more tractable problem.

We can use things like Multi-version Concurrency Control (MVCC) to detect conflicts at write-time and abort or re-run the transaction based on the new state.

Handling concurrent writes like this is in a way that leads to tractable, consistent state is (as we discussed last week), one of the hardest problems to solve in distributed systems. And, as Blah Blah the CTO of Convex puts it, "There's no amount of getting smarter that allows you easily solve concurrent offline writes".

That means there's no CRDT silver bullet. There is no "better math" that can deterministically lead to a merged state that users don't ever interpret as data-loss.

As Clemmer, one of the leading CRDT researchers puts it:

> Offline editing is a UI/UX problem, not an algorithms problem.

The issue isn't that CRDTs produce bad merges. It's that CRDTs lead us to believe we can have offline-writes **and** avoid conflicts.

We can't.

## Part 2: What to do

Given the limitations of conflict-avoidant software, there are two reasonable responses.

**Path 1: Use a server as the authority.** Moment's [second post](https://www.moment.dev/blog/lies-i-was-told-pt-2) advocates for this approach, demonstrating that a ~40 line implementation using `prosemirror-collab` achieves the same user experience as a CRDT for most use cases:

> I have heard the argument more times than I can count: CRDTs are operationally complex, but you need them (_need_ them!) for optimistic updates, edits during network blips, fine-grained provenance of edits, peer-to-peer reconciliation, and so on. I want to convince you that _all_ of these things (except true masterless p2p architecture) are easily doable without CRDTs.

In this model, the server is the single source of truth. Clients send operations, the server rebases them against concurrent edits, and clients accept whatever the server says. It's simple, debuggable, and avoids the weird merge semantics entirely — conflicts are resolved by operation ordering, not by algorithmic merge.

The tradeoff is that you need a server, and the server must be available for edits to be accepted. True offline editing becomes "optimistic local edits that might be rejected when you reconnect."

**Path 2: Keep the CRDT properties, but surface conflicts.** This is closer to what we're building with ebb.

Matthew Weidner, who spent years researching text-editing CRDTs, [recently argued](https://mattweidner.com/2025/05/21/text-without-crdts.html) that you don't need a complex CRDT library to get CRDT-like properties:

> The main issue with both CRDTs and OT is their _conceptual complexity_. Text-editing CRDTs' total orders are subtle algorithms defined in academic papers, often challenging to read. [...] Complicated algorithms lead to even more complicated implementations, and CRDT/OT implementations are notoriously difficult.

His alternative? Label each character with a unique ID, send "insert after" operations that reference those IDs, and use a deterministic ordering rule. The same convergence guarantees, achieved with straightforward code you can actually understand and modify.

This is essentially what we've built. And to be completely honest: **our causal tree would produce the same "u" result in the Colour scenario**. The merge semantics are identical to a traditional CRDT — we're not claiming to have solved that problem algorithmically.

What we _are_ claiming is that the merge result shouldn't be the end of the story.

## Conflicts as a feature, not a bug

CouchDB figured this out years ago. When concurrent edits create a conflict, CouchDB doesn't pick a winner and discard the loser. It picks a _deterministic_ winner (so all replicas converge), but it _stores both versions_ and exposes them through an API. Applications can then surface those conflicts to users, merge them intelligently, or handle them however makes sense for their domain.

This is the right mental model: **deterministic convergence for the machines, conflict surfacing for the humans**.

A collaborative editing system should have:

**Convergence.** All replicas must eventually reach the same state. This is what CRDTs provide, and it's genuinely valuable. We're not throwing this out.

**Transparency.** When concurrent edits produce an unexpected result, you should be able to inspect exactly what happened and why. The merge logic shouldn't be buried in a library you can't read.

**Conflict detection.** The system should know when a merge happened that might have produced surprising results. Not every concurrent edit is a conflict — two users typing in different paragraphs is fine. But when Alice deletes text that Bob was editing, that's worth flagging.

**Conflict surfacing.** Rather than pretending conflicts don't exist, surface them to users when they occur. A human seeing "Alice deleted this heading while you were fixing a typo — what do you want to do?" is infinitely better than silently producing a document containing just "u".

**Ergonomic levers.** Developers need tools to decide _how_ conflicts are handled for their specific use case. Some conflicts should be auto-resolved. Some should be surfaced to users. Some should block until resolved. The framework shouldn't dictate a one-size-fits-all policy.

## How ebb approaches this

Ebb doesn't use a CRDT library, but we _do_ use CRDT-like data structures. The demo below builds a collaborative text editor from scratch using a **Causal Tree** ordered by **Hybrid Logical Clocks**. This is functionally equivalent to the RGA CRDT — we get the same convergence guarantees, including the same "u" result in pathological cases.

The difference is what happens next.

When ebb detects that concurrent edits touched the same data, it doesn't just merge and move on. It snapshots the pre-merge state and surfaces the conflict through an API — similar to CouchDB's approach. Developers can then decide whether to show the user a diff, auto-resolve based on business rules, or let the merge stand silently.

## Identity over indices

The core problem with collaborative text editing is deceptively simple: when two users type concurrently, array indices become meaningless. Alice inserts at position 3, shifting everything after it — but Bob's concurrent insert at position 17 doesn't know that yet.

The fix: stop thinking in indices. Give each run of characters a stable ID and reference neighbors instead. `insert after node X` works no matter what else has changed. These parent references form a **Causal Tree** — sequential typing creates chains, concurrent edits at the same position create branches. Reading in depth-first order reconstructs the document deterministically.

Run-length optimization means sequential characters by the same peer are stored as a single node, keeping the tree compact even for long documents.

## Order without authority

A tree of character runs solves the identity problem, but you still need a rule for ordering siblings — when two users type at the exact same position, who goes first?

CRDTs typically solve this with complex, algorithm-specific total orders. We use a **Hybrid Logical Clock**: a wall-clock timestamp combined with a monotonic counter that guarantees every operation gets a unique, totally-ordered ID. The ordering rule is deterministic — peers converge to the same result regardless of whether a server relays the operations or they arrive peer-to-peer.

When a remote operation arrives, the peer merges the remote clock with its own, ensuring time only moves forward. The tie-break rule — higher HLC wins — is what makes convergence deterministic without any central coordination.

## Sync you can inspect

One of the strongest critiques of CRDT libraries is that they're opaque — operations go in, state comes out, and when something goes wrong, you can't see why. Our wire protocol is fully transparent: every edit produces an **Action** containing self-describing **Updates**, each targeting a specific run node with a named method and typed field data.

- **PUT** inserts a new run into the tree
- **PATCH** appends characters to an existing run
- **DELETE** tombstones a character range within a run

A multi-character delete produces one Action with multiple DELETE Updates — not separate messages. You can read every operation, trace every mutation, and understand exactly why the document looks the way it does.

## What's next

This demo is a proof of concept, but it demonstrates the core philosophy behind ebb's approach: **we're not trying to eliminate conflicts — we're trying to make them manageable**.

The same causal tree structure shown here will power real-time collaboration in ebb. In production, the relay won't be a `BroadcastChannel` between two browser tabs — it'll be ebb's sync engine: Actions flowing through a durable log, fanned out to subscribers over SSE, with the same HLC-based ordering ensuring every client converges to the same state.

But convergence is just the foundation. When concurrent edits produce results that might surprise users — like the "Colour" scenario — ebb [detects and surfaces those conflicts](/docs/conflicts) rather than silently merging them. Developers get ergonomic levers to decide how conflicts are handled: surface them to users, auto-resolve based on rules, or let the merge stand.

The CRDT community got the hard part right: stable identities, causal ordering, deterministic merge rules. What the _ecosystem_ got wrong was treating "conflict-free" as a feature instead of a tradeoff. Humans don't work in a conflict-free way. We disagree, we make concurrent decisions, we change our minds. Software that hides this reality isn't helping — it's lying.

The answer isn't to abandon CRDTs for simpler server-authoritative models (though that's a valid choice for many applications). The answer is to stop building conflict-avoidant software and start building conflict-embracing software. Let the machines converge deterministically. Let the humans decide what that convergence should mean.
